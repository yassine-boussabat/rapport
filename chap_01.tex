\section{Longview}
% ─────────────────────────────────────────────

\begin{figure}[h!]
\centering

\caption{Linedata Longview — Front-Office Trade Order Management Platform}

\paragraph{} Linedata Longview is a fully integrated, multi-user front-office trade order management solution made for buy-side institutions. It helps portfolio managers and traders with all of their business needs and workflows throughout the entire investment lifecycle, from building portfolios and checking for compliance to making orders, trading electronically, and processing trades after they happen. \cite{longview_guide}

\paragraph{}According to its official documentation, Longview offers a comprehensive suite of capabilities built around an open relational database architecture that supports 24×7 global trading and straight-through processing. Its framework is designed to integrate seamlessly with accounting systems, settlement platforms, data vendors, and other third-party applications. \cite{longview_guide} The platform is currently at version 8.2 and is actively maintained and extended by Linedata's engineering teams.

\subsection{Core Capabilities}

\paragraph{}Longview's functional scope covers the following major areas of the investment management workflow:

\begin{itemize}
    \item \textbf{Portfolio Management and Appraisals:} Portfolio managers use Longview's Appraisal reports to review account holdings, monitor positions and proposed orders, and evaluate relative weightings. The platform supports both single-account appraisals and group appraisals that consolidate holdings across user-defined groups of accounts. \cite{longview_guide}
    \item \textbf{Modeling and Rebalancing:} Longview supports multiple model types — Portfolio Models, Sector Models, Self Models, Variance Models, and Model of Models — to represent investment strategies. Portfolio rebalancing generates the necessary pro forma orders to bring account holdings in line with model targets, with configurable rounding rules, minimum trade quantities, and drift tolerance ranges. \cite{longview_guide}
    \item \textbf{Order Management (OMS):} The platform provides full order lifecycle management. Portfolio managers create pro forma orders — ``what if'' orders that simulate the effect of trades on portfolio weightings and cash — which are then sent to traders for execution after passing compliance checks. Longview supports individual orders, group orders, and advanced order types including debt securities, repos, TBAs, NDFs, swaps, and CFDs. \cite{longview_guide}
    \item \textbf{Electronic Trading and FIX Connectivity:} Longview supports electronic trading via FIX to automate and streamline trading activities, including placement routing, algorithmic trading, and multi-leg trade processing. It integrates with broker systems through the Trader Blotter and supports broker ticket management from creation through confirmation and settlement. \cite{longview_guide}
    \item \textbf{Investment Compliance:} Pre-trade and post-trade compliance checks are run against account-level investment restrictions and regulatory mandates. Compliance officers can manage, override, and audit breach events with full traceability. \cite{longview_guide}
    \item \textbf{Cash and Risk Management:} Longview enables real-time management of cash balances at the portfolio and sector level, with support for multi-currency portfolios and FX hedging. Risk exposures across counterparties, sectors, and currencies are monitored through configurable analytics dashboards. \cite{longview_guide}
    \item \textbf{Reporting and Audit:} The platform provides rich reporting through the Manager Blotter, Order Preview, Security Cross-Reference, Ticket Summary, and Placement Blotter reports. Complete audit trail reports are available for orders, executions, allocations, and settlement tickets. \cite{longview_guide}
\end{itemize}

\subsection{User Roles and Access Control}

\paragraph{}Longview operates with a role-based access control model. Pre-loaded roles include Administrators, Portfolio Managers/Traders, Compliance Officers, Traders, Trading Assistants, Developers, and several specialized administrative roles. Each role is granted a specific set of permissions covering data creation, updates, and read access across the platform's entities (accounts, securities, orders, brokers, models, etc.). \cite{longview_guide}

\subsection{Core Data Entities}

\paragraph{}Longview's underlying SQL Server database is organized around a set of interdependent financial entities that must maintain strict referential integrity and business-rule consistency at all times. These entities are the primary targets of the test data generation system developed in this project:

\begin{itemize}
    \item \textbf{Securities:} Financial instruments including equities, fixed-income (amortizing, non-amortizing, discount, floating rate), derivatives (futures, options, swaps, NDFs), FX, repos, TBAs, and generic securities. Each security is identified by a set of market codes (ISIN, SEDOL, CUSIP, ticker). \cite{longview_guide}
    \item \textbf{Accounts and Account Groups:} Portfolio accounts organized in hierarchical workgroups, each associated with default models, hierarchies, rebalancing rules, and compliance restrictions. Accounts may belong to multiple account groups. \cite{longview_guide}
    \item \textbf{Models and Hierarchies:} Definitions of portfolio structure expressed as weighted targets for securities or sectors. Models drive the rebalancing logic and are tightly linked to account configurations.
    \item \textbf{Orders:} Buy and sell instructions for securities, associated with accounts, trade directions, quantities, prices, time-in-force rules, and compliance status. Orders progress through a lifecycle from pro forma to confirmed via the trading workflow. \cite{longview_guide}
    \item \textbf{Executions:} Records of actual trade fills entered via the Fast Fill window or Broker Ticket, linked to parent orders and brokers, including quantity, price, and settlement data. \cite{longview_guide}
    \item \textbf{Positions:} Real-time and historical holdings of securities within accounts, computed from executions and adjusted through corporate actions and settlements. \cite{longview_guide}
    \item \textbf{Prices:} Market prices and valuation data for securities, consumed as streaming data from external feeds or imported via price files. Prices are used in appraisals, rebalancing calculations, and compliance evaluations. \cite{longview_guide}
\end{itemize}

\paragraph{}The relationships between these entities are governed by a rich set of financial and referential constraints. For example, an execution must reference a valid parent order; the sum of execution quantities must not exceed the parent order quantity; position values must be consistent with the cumulative effect of all associated trades; and price data must exist for a security before it can be correctly valued in an appraisal. This intrinsic complexity makes the creation of valid and realistic test datasets a particularly challenging task.

\section{Current Testing Procedure}
% ─────────────────────────────────────────────

\paragraph{}Ensuring the correctness and reliability of Longview's features across every software release requires a robust and continuous testing strategy. The QA and development teams at Linedata are responsible for validating new features, performing regression testing, and ensuring that existing functionality remains intact as the platform evolves. This process depends fundamentally on the availability of relevant, varied, and realistic test data.

\subsection{Manual Test Data Preparation}

\paragraph{}The current approach to test data creation is largely manual. QA engineers and developers must craft SQL INSERT scripts by hand, targeting the Longview SQL Server database directly. For each test scenario — whether it covers a complete trade lifecycle, a compliance breach, a multi-asset portfolio rebalancing, or a group order allocation — the tester must:

\begin{enumerate}
    \item Identify all database tables involved in the scenario (securities, accounts, models, orders, executions, positions, prices, and their dependency tables).
    \item Manually construct valid records for each entity, ensuring that all primary and foreign key relationships are satisfied and that auto-generated identifiers are correctly cross-referenced.
    \item Verify that the generated data respects the business rules enforced by the Longview application layer — such as quantity consistency between orders and executions, valid price ranges for a given security type, and correct account-model associations.
    \item Execute the SQL scripts against a test database instance, validate the application's behavior, and iterate to correct any data inconsistencies detected during the test run.
\end{enumerate}

\paragraph{}This process demands substantial domain expertise — a solid understanding of both the SQL Server schema structure and the financial logic governing each entity. Even experienced team members spend considerable time constructing datasets that correctly reflect real-world financial scenarios.

\subsection{Limitations of the Current Approach}

\paragraph{}The manual approach to test data creation presents several significant limitations that reduce the efficiency, coverage, and reliability of the QA process:

\begin{itemize}
    \item \textbf{Time-consuming:} Building a coherent dataset across multiple interdependent tables — securities, accounts, models, orders, executions, positions, and prices — requires significant effort. Complex scenarios such as a full trade lifecycle involving group orders, multi-account allocations, and broker tickets can take hours to set up manually.
    \item \textbf{Error-prone:} Human errors in data entry — mismatched foreign keys, incorrect security type attributes, execution quantities exceeding parent order quantities, or missing price records — frequently cause test failures that are unrelated to the feature under test, leading to wasted debugging effort.
    \item \textbf{Limited scenario coverage:} The manual process naturally produces a small number of test datasets, leaving large parts of the application behavior untested, particularly edge cases, boundary conditions, and less common financial instrument types such as repos, TBAs, swaps, and NDFs.
    \item \textbf{Lack of reproducibility:} Test data is typically created ad hoc and not stored in a structured, version-controlled format. This makes it difficult to reproduce historical test scenarios, share datasets between team members, or reuse them across regression cycles.
    \item \textbf{Confidentiality constraints:} Production data from real clients cannot be used for testing due to strict data confidentiality agreements and regulatory requirements. This limits access to the most realistic datasets and forces teams to rely entirely on artificial data.
    \item \textbf{Environment dependency in connected mode:} When tests require interaction with a live SQL Server instance, there is a dependency on the availability of a correctly configured database environment, which introduces additional setup overhead and potential instability.
\end{itemize}

\section{Problematic}
% ─────────────────────────────────────────────

\paragraph{}The limitations described in the previous section reveal a clear and pressing gap in Linedata's testing infrastructure. As the Longview platform grows in scope — supporting an ever-broader range of financial instrument types, regulatory compliance frameworks, and client configurations — the demand for varied, realistic, and rapidly deployable test data grows proportionally. The current manual approach, built around individual expertise and ad hoc scripting, is no longer sustainable at the pace required by modern continuous-integration software delivery.

\paragraph{}This situation leads us to formulate the central problematic of this project as follows: \textit{how can Linedata's development and quality assurance teams be equipped with an automated, configurable, and domain-aware tool that generates realistic, coherent, and SQL-ready test data for the Longview platform, while correctly enforcing the complex inter-entity relationships and financial business rules that govern its data model?}

\paragraph{}This general question unfolds into several concrete technical challenges that must be addressed by the proposed solution:

\begin{itemize}
    \item \textbf{Financial data complexity and consistency:} Longview's data model involves dozens of interdependent tables, and generating valid test data requires simultaneously enforcing hundreds of business rules — for example, ensuring that execution quantities do not exceed their parent order quantities, that position values remain coherent with their associated trade history, that price records exist for every valued security, and that account-model associations are structurally valid. Any violation renders the dataset unusable.

    \item \textbf{Dual-mode operation:} The tool must function in two distinct modes. In \textit{standalone mode}, it generates fully synthetic data without requiring any external database connection, making it usable in isolated development environments. In \textit{connected mode}, it establishes a connection to an existing Longview SQL Server instance, retrieves real reference data (securities, accounts, models), and generates additional records that are consistent with the existing state of the database. Each mode imposes different architectural and security constraints.

    \item \textbf{Multi-asset instrument support:} Longview handles a wide variety of financial instrument types — equities, fixed-income bonds (amortizing, non-amortizing, discount), derivatives (futures, options, swaps, NDFs), repos, TBAs, FX, and generic securities. \cite{longview_guide} Each type has its own set of mandatory fields, valid value ranges, and pricing conventions. The generation logic must account for these per-instrument constraints.

    \item \textbf{SQL script export:} The generated data must be exportable as ready-to-execute SQL INSERT scripts compatible with the exact Longview SQL Server schema. This requires precise mapping between the domain model used internally by the generation engine and the underlying database table structure, including correct handling of identity columns, datetime formats, and nullable fields.

    \item \textbf{Configurability and usability:} Test engineers must be able to specify the volume, type, date ranges, and characteristics of the data to be generated through an intuitive web-based interface — without writing SQL or modifying source code. The tool must be accessible to both technical developers and less technical QA profiles.

    \item \textbf{Extensibility and optional LLM integration:} As financial data conventions evolve and new instrument types are introduced, the tool must be architecturally extensible. Additionally, a large language model (LLM) integration layer should optionally enrich the generation process with more semantically varied and contextually appropriate data fields, such as security names, ISIN codes, account labels, and strategy descriptions.
\end{itemize}

\paragraph{}Addressing these challenges requires the design and implementation of a full-stack application combining a React + TypeScript frontend for configuration, a NestJS backend with domain-aware generation services for each financial entity, and a robust SQL Server integration layer for the connected mode. This is the system described in the remainder of this report.
