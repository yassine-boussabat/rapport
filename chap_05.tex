\subsection{Analysis and Verification}

For the optimal LTS-based testing approach, the analysis and verification involve the following key steps:

\subsubsection{Generation of the Regular Expression}
This step involves creating a regular expression that accurately represents the set of all possible paths in the labeled transition system (LTS). The generated regular expression serves as a comprehensive model of the system's behavior, enabling us to identify and analyze various path combinations.

We conducted several tests on the algorithm for generating regular expressions and found that it is generally efficient, producing correct regular expressions from the graph in most cases. However, there is a specific scenario where the algorithm fails to handle the case correctly. Through manual testing, we identified that the algorithm does not address this particular situation.

    \centering

    \label{fig:ExampleRegex}
\end{figure}


In fact, the issue arises when the algorithm encounters a node that has already been processed earlier in the traversal. Specifically, when the algorithm revisits a node that was previously treated, it does not correctly generate the paths that should include this revisited node. For instance, in the provided graph (as shown in Figure \ref{fig:ExampleRegex}), the path from \(S0 \rightarrow S3 \rightarrow S7 \rightarrow S5\) should include the sub-path that revisits \(S5\). However, the current algorithm does not account for this, leading to an incomplete regular expression generation. The generated regular expression for this example is:
\[
((abc)^{*}(bcc)^{*})^{*}d(fae)^{*} + ((abc)^{*}(bcc)^{*})^{*}bdf_
\]

From this regular expression, we cannot deduce the path corresponding to the sequence "
bdee",  which should be included in the regular expression.

The underlying issue in the algorithm stems from its handling of previously visited nodes. When the algorithm detects that a node is already in the stack or has been processed (i.e., is present in the \texttt{tas} set), it prematurely terminates further exploration for that node, returning an incomplete language that doesn't account for the revisited paths.

This problem can be seen in the provided algorithm for regular expression generation. The algorithm correctly identifies and processes loops within the same node (using the \texttt{countCircuits} variable), but it fails to correctly handle cases where a node is revisited later in the traversal. Specifically, when a node like \(S7\) in the example graph points back to a previously processed node \(S5\), the algorithm should continue processing this connection to correctly integrate the revisited path into the final regular expression. Unfortunately, the current implementation stops further exploration for such revisited nodes, leading to the omission of paths like "bdee" in the final regular expression.

Addressing this issue requires modifying the algorithm to properly handle the revisited nodes by ensuring that paths including these nodes are fully explored and integrated into the resulting regular expression. This modification would allow the generation of all valid paths, including those that revisit nodes previously encountered in the traversal.



\subsubsection{Extraction of Paths}
In this phase, we extract all feasible paths from the regular expression. This extraction process is crucial for understanding the paths that the system can potentially take and for evaluating the completeness and efficiency of the generated paths.

While testing this functionality, we discovered that the algorithm initially generates the correct paths from the regular expression. However, we identified an additional operation that introduces redundancy. Specifically, the algorithm appends subpaths from the already generated paths, which leads to unnecessary paths being included and negatively impacts performance.  Specifically, the algorithm appends subpaths from the already generated paths during the phase where the function \textbf{extractPaths()} is used to process the paths extracted from the regular expression.

The relevant part of the code involves the extraction of paths, where the function dealing with paths generation is called within a loop to process all paths from the regular expression. After extracting the paths, they are stored in \textbf{`generatedPaths`}, which is then further processed to eliminate null paths. However, during this process, the code inadvertently adds subpaths that are not necessary for the problem at hand.

\subsubsection{Application of Linear Programming}
The final step involves applying linear programming techniques to optimize the selection of
paths. Linear programming helps determine the optimal set of paths that cover all transitions
in the LTS while minimizing the number of paths and their lengths. This optimization ensures
efficient and comprehensive testing of the system.

\textbf{Problem 1: Constraints matrix dimensions}\\ We observed that the number of columns in the constraints matrix is equal to the sum of the number of paths and the number of transitions, which is not expected. In the code, this is represented by the following line:\\
\texttt{ \[constraints [g->transitions.size()][generatedPathsnnull.size() + g->transitions.size()]; \]} 

However, this is incorrect because, the dimensions of the constraints matrix should be \textit{nb transitions} by \textit{nb paths}. This discrepancy in matrix dimensions introduces unnecessary complexity and could potentially lead to incorrect results during the linear programming optimization process. The inclusion of additional columns, which correspond to the number of transitions, is redundant and does not align with the intended purpose of the constraints matrix.

The correct matrix should only have as many columns as there are paths, with each row corresponding to a transition. Each element in the matrix should indicate whether a particular path covers a given transition. The extra columns added due to the erroneous code might create an incorrect formulation of the linear programming problem, which in turn could affect the performance and accuracy of the path selection process.

This issue needs to be corrected to ensure that the matrix accurately reflects the relationship between paths and transitions, thereby enabling the linear programming solver to find the optimal set of paths without unnecessary overhead.

\textbf{Problem 2: Wrong ILP Model Implementation}\\
The implementation of the linear programming (LP) model contains several errors that affect its correctness and efficiency. 

Firstly, the number of variables created for the LP model is incorrect. The implementation mistakenly considers the number of variables as the sum of the number of feasible paths and the number of transitions. However, the correct approach should only account for the number of feasible paths. This leads to an unnecessary increase in the number of variables, which complicates the optimization process.

Moreover, there is an issue with how the coefficients for the objective function are calculated. The goal of the LP model is to minimize the total length of the paths. In the correct modeling of the ILP, as mentioned in Chapter 4, the profit coefficients should represent the lengths of the paths. Upon executing the solution, we discovered that the coefficients were not correctly calculated. Instead of calculating the length of the paths by summing the number of transitions, the algorithm mistakenly used the length of the string representing the path. This miscalculation leads to incorrect objective function values, which negatively affects the optimization process. As a result, the LP model does not accurately represent the intended problem, undermining the effectiveness of the path selection process.

\subsubsection{Generation of the Regular Expression}
In this section, we present the correct algorithm for generating the regular expression that represents the language of the system. We begin by introducing some notations and the data structures used before describing the proposed algorithm. \\

\textbf{Notations and data structures}\\
Given a state $s$, a set of alphabets $\Sigma$, $Reg(\Sigma)$ denotes the set of regular expressions built on the elements of $\Sigma$. \\Also,
\begin{itemize}
    \item $Lg(s) = \{<seq,s'> \mid seq \in Reg(\Sigma)$, and s' is the id of a state or$ \, \emptyset\}$
    \item  Given a sequence $\sigma \in \Sigma^{*}$, $\sigma.Lg(s)=\{<\sigma.seq,s'>\mid <seq,s'> \in Lg(s)\}$ %\textcolor{red}{a voir}
    \item $Cy(Lg(s))$ denotes the set of elements $e\in Lg(s)$ s.t. $e.state=s$. %\textcolor{red}{a voir} 
    \item $\overline{Cy(s)}=Lg(s)\setminus Cy(s)$. 
\end{itemize}

Algorithm~\ref{alg:regexgen} is based on a depth-first, recursive traversal of the graph associated with the system under test. The algorithm takes as input a stack, initially containing only the initial state. Each state encountered during the traversal is either fully processed and stored in a map or partially processed and retained in the stack for further treatment.

If a successor is not in the current language but has been visited, the function \texttt{UpdateLanguage()} is called (developed in Algorithm ~\ref{alg:update}) to update or compute the language of this successor before updating the language of the current state. This is an important case, as illustrated in Figure \ref{fig:ExampleNewExec}. We encounter this situation in two instances:  
(a) At state \(s_3\), label \(c\) leads to state \(s_6\), which has already been fully processed and removed from the stack.  
(b) At state \(s_7\), label \(e\) leads to state \(s_5\), which has also been fully processed and is no longer on the stack.

\subsubsection{Application of Linear Programming}
We proposed a refined formalization of the Integer Linear Programming (ILP) model, ensuring that the code implementation faithfully adheres to the theoretical framework. The key components are defined as follows:

\begin{itemize}
    \item \textbf{Variables:} Each variable \( x_j \) represents a path in a given Labeled Transition System (LTS).
    
    \item \textbf{Coefficients:} The coefficient \( a_{ij} \) indicates the number of occurrences of the transition label \( t_i \) in the path \( x_j \). Unlike the previous approach, where \( a_{ij} \) simply denoted the presence or absence of a transition in a path, the new formulation captures the exact count of occurrences. If \( a_{ij} > 1 \), it indicates that the path \( x_j \) contains multiple occurrences of the label \( t_i \); otherwise, the label is absent in the path.
    
    \item \textbf{Constraints:} Each constraint corresponds to a transition label in the LTS. The number of constraints is equal to the number of LTS labels, and all constraint expressions must satisfy \( \geq 1 \), ensuring that each label is covered by at least one path.
    
    \item \textbf{Objective Function:} Every objective coefficient \( c_i \) represents the length of a path, calculated as the sum of the constraint coefficients for that path, i.e., \( c_i = \sum_{j=1}^{m} a_{ji} \). The objective function aims to minimize the total coverage cost by selecting the shortest paths that cover all transitions in the LTS.
\end{itemize}

\begin{mdframed}[leftmargin=15pt, rightmargin=15pt]
\begin{equation*}
\begin{array}{lr@{}c@{}r@{}l@{}r}
\text{\textbf{Minimize}}   & c_1 x_1   & {}+{\dots}+{} & c_n x_n   &        &   \\
\text{\textbf{Subject to} } & a_{11}x_1 & {}+{\dots}+{} & a_{1n}x_n & {}\geq 1   & \\
                   &           &     \vdots    &               & \hspace{1.5cm} x_{j} \in \{0,1\}, j=1 ,..., n   \\
                   & a_{m1}x_1 & {}+{\dots}+{} & a_{mn}x_n & {}\geq 1 & 
\end{array}
\end{equation*}
\end{mdframed}

\section{Extension}
The goal of this extension is to address the challenges of scaling in large Labelled Transition Systems (LTS), where traditional linear programming methods become insufficient as the size of the Symbolic Observation Graph (SOG) increases, particularly in terms of the number of states to be explored. Consequently, we aim to leverage deep learning techniques to guide the exploration of state spaces in large systems with an excessively large number of accessible states.

Initially, when the size of the graph remains manageable, an exact solution is employed to explore and construct the SOG, ensuring precise results. During this exploration, a crucial step is the generation of a dataset, which captures key features and patterns. This dataset will later be used to train a machine learning (ML) model.

As the SOG grows and surpasses a certain threshold, relying solely on exact methods becomes impractical. In these scenarios, the ML model takes over, guiding the exploration process by analyzing the prefix of the current nodes and the possible future transitions, and recommending the best paths to follow. This strategy aims to optimize state space exploration, reduce computational complexity, and improve efficiency in large-scale LTS environments.

In this section, we will focus on the crucial step of dataset generation and present the steps we followed to achieve our goal. By meticulously generating and curating this dataset during the initial exploration phase, we ensure that the machine learning model is well-equipped to guide the exploration process effectively, even as the system grows in complexity.

To effectively generate the dataset, it is essential to first derive the set of optimal paths obtained through the linear programming solution, as these paths will guide the data collection process. 

The process begins with a detailed traversal of the graph, starting from the initial node. During this traversal, we systematically generate all possible prefixes of the graph. These prefixes form the foundational elements of the dataset, encapsulating the initial sequences of transitions that the system may undergo.

Once the prefixes are established, we proceed with a breadth-first search (BFS) across the graph to expand the dataset. This BFS approach allows us to explore the graph systematically, ensuring that all relevant state transitions are considered. For each prefix encountered, we carefully identify the possible successors—these represent the various choices or transitions available at each stage of the exploration. By projecting these potential successors onto the previously derived set of optimal paths, we can determine the most favorable or optimal choice for each specific scenario. This method not only enriches the dataset with valuable decision points but also ensures that the dataset is aligned with the optimal exploration strategy, thereby enabling the machine learning model to make precise and efficient decisions when navigating large and complex state spaces.

To further illustrate the concept of projection onto the optimal solution, let's consider an example from Figure \ref{fig:ExamplePaths}. In the previous section, we extracted the set of optimal paths: \(adeag\) and \(abfc\). Now, let's take a specific prefix from the graph, such as \(adea\). At this point in the graph, as shown in Figure \ref{fig:exampleProjection1}, the possible transitions from \(adea\) are \(b\), \(d\), and \(g\).

\centering
\resizebox{0.7\textwidth}{!}{\input{img/example_projection1}}
\caption{Example 1 of the best choice from a prefix \label{fig:exampleProjection1}}
\end{figure}


The goal of the projection onto the optimal solution is to determine the best choice among these transitions. This determination is made by comparing the available transitions to the set of optimal paths we previously identified. In this example, when we project the prefix \(adea\) onto the optimal paths \(adeag\) and \(abfc\), we see that the transition \(g\) aligns with the optimal path \(adeag\). Therefore, the best choice at this point is to follow the transition \(g\).

\centering
\resizebox{0.7\textwidth}{!}{\input{img/example_projection2}}
\caption{Example 2 of the best choice from a prefix \label{fig:exampleProjection2}}
\end{figure}

Now, let's consider another prefix as an example, such as \(abea\), as shown in Figure \ref{fig:exampleProjection2}.

At this stage, the possible transitions are \(d\), \(b\), and \(g\). However, when we project this prefix onto the set of optimal paths, we find that none of the transitions continue the sequence in a way that aligns with either \(adeag\) or \(abfc\). In this case, the projection indicates that none of the available transitions represent an optimal continuation. This means that, for the prefix \(abea\), there is no best choice according to the optimal paths, as the sequence of the prefix does not appear in any of the optimal paths.

This example demonstrates how projection onto the optimal solution helps in decision-making during the exploration of state spaces. By analyzing the relationship between the current prefix and the optimal paths, we can determine the most effective transitions to follow, or recognize when no optimal transition is available. This approach ensures that the exploration remains aligned with the optimal paths, enhancing the efficiency and accuracy of the testing process.
