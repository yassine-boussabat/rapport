\section{Development Methodology}
% ─────────────────────────────────────────────

\paragraph{}Given the exploratory and iterative nature of this project — building a domain-specific tool with requirements that sharpen as understanding of the target platform deepens — an agile approach was the natural choice. Among the available agile frameworks, Scrum was selected for its lightweight structure, its emphasis on short delivery cycles, and the way it accommodates changing requirements without derailing the overall plan.

% ─────────────────────────────────────────────
\subsection{Scrum Framework Overview}
% ─────────────────────────────────────────────

\paragraph{}Scrum organizes work into fixed-length iterations called sprints, each ending with a potentially shippable increment of the product. Within this project, the Scrum roles were adapted to the internship context as follows:

\begin{itemize}
    \item \textbf{Product Owner:} The internship supervisor at Linedata, responsible for defining priorities, validating deliverables, and ensuring the tool aligned with the team's actual testing needs.
    \item \textbf{Scrum Master / Developer:} The intern, responsible for planning, executing, and delivering each sprint increment, as well as managing impediments and adjusting scope when needed.
\end{itemize}

\paragraph{}The main Scrum artifacts used throughout the project were the \textbf{product backlog} — a prioritized list of all features and tasks — and the \textbf{sprint backlog} — the subset of items committed to for each sprint. Brief review sessions at the end of each sprint served as the delivery checkpoint, where the increment was demonstrated and feedback was collected before planning the next cycle.

\section{Study of Existing Solutions}
% ─────────────────────────────────────────────

\paragraph{}Before designing anything, it made sense to look at what already exists in the space of test data generation. Several tools and approaches address this problem in general terms. Understanding where they fall short in this specific context helped justify the decision to build a custom solution.

% ─────────────────────────────────────────────
\subsection{Generic Test Data Generation Tools}
% ─────────────────────────────────────────────

\paragraph{}The most commonly used tools for generating synthetic test data fall into three categories:

\begin{itemize}
    \item \textbf{Mockaroo} \cite{mockaroo}: A web-based tool that generates random data across a wide range of field types and exports it in various formats including SQL, CSV, and JSON. It is quick to use for simple flat datasets but has no concept of relational integrity or cross-table dependencies. There is no way to encode the kind of rules that Longview requires — for instance, ensuring that every position references a security that has a corresponding price record, or that execution quantities never exceed the parent order.

    \item \textbf{Faker (various language libraries)} \cite{faker}: A widely used library available in Python, JavaScript, and other languages that generates plausible fake data — names, addresses, numbers, dates. It is excellent for populating individual fields with realistic-looking values, but it operates at the field level, not the dataset level. Building coherent multi-table financial datasets on top of Faker requires writing all the coordination logic by hand, which is essentially the same problem the tool was meant to solve.

    \item \textbf{SQL Server Data Tools / Redgate SQL Data Generator} \cite{redgate}: These tools are closer to the target use case — they generate SQL INSERT statements for relational schemas and can respect foreign key relationships. However, they operate on a generic schema inference basis. They have no knowledge of financial domain rules, no concept of Longview's pre-BCP buffer mechanism, and no way to produce the specific Unicode and type casting conventions the platform requires. The scripts they produce would need significant manual post-processing before they could be used with Longview.
\end{itemize}

% ─────────────────────────────────────────────
\subsection{Why None of Them Fit}
% ─────────────────────────────────────────────

\paragraph{}The gap these tools all share comes down to the same thing: they are built for the general case, not for a specific financial platform with its own data model, its own loading conventions, and its own business rules. Longview's pre-BCP buffer architecture, the \texttt{N'...'} Unicode prefix requirement, the typed numeric identifiers, the cross-entity constraints between orders, executions, positions, and prices — none of this can be encoded in a generic tool without effectively rebuilding the domain logic from scratch. At that point, you are no longer configuring an existing tool; you are building a custom one anyway.

\paragraph{}This analysis confirmed that a purpose-built solution was the right path, and that the core value of the tool would lie precisely in the domain knowledge encoded into its generation logic.

\subsection{Functional Requirements}
% ─────────────────────────────────────────────

\paragraph{}The functional requirements were gathered through analysis of the existing SQL scripts, discussions with the Longview development team, and review of the project brief. They fall naturally into two groups, corresponding to the two operating modes of the system.

\subsubsection{Mode 1 — Parameter-Driven Generation (Standalone)}

\paragraph{}In this mode, the system generates a complete, self-contained test dataset from user-supplied parameters, with no database connection required. The specific requirements are:

\begin{itemize}
    \item The user must be able to specify the number of accounts, securities per asset class, orders, executions, and positions to generate.
    \item The user must be able to configure date ranges, base currencies, instrument types, and scenario-level options through a web interface.
    \item The system must generate all entities in dependency order — securities before accounts, accounts before orders, orders before executions and positions.
    \item All generated data must respect Longview's pre-BCP buffer conventions: \texttt{N'...'} string literals, \texttt{CAST(N'...' AS DateTime)} date values, and \texttt{CAST(n AS Numeric(10,0))} identifiers.
    \item The output must be a complete, executable SQL script that can be run on any clean Longview SQL Server instance without modification.
    \item The system must support all major asset classes handled by Longview: equities, fixed income, futures, options, and FX forwards.
\end{itemize}

\subsubsection{Mode 2 — Existing Data Generation (Connected)}

\paragraph{}In this mode, the system connects to a live Longview SQL Server instance in read-only mode, reads what is already there, and generates additional records that are coherent with the existing database state. The specific requirements are:

\begin{itemize}
    \item The user must be able to provide SQL Server connection credentials through the interface.
    \item The system must retrieve existing accounts, securities, models, and prices from the connected instance.
    \item Generated orders, executions, and positions must reference only entities that actually exist in the connected database.
    \item The connection must be strictly read-only — the application must never write to, modify, or delete any data in the connected database.
    \item The output is still a SQL script, not a direct database write — the user runs it manually after review.
\end{itemize}

\subsection{Non-Functional Requirements}
% ─────────────────────────────────────────────

\paragraph{}Beyond what the system must do, several quality constraints shape how it should behave:

\begin{itemize}
    \item \textbf{Performance:} Generating a dataset of typical size — covering one account with a full set of securities, orders, executions, and positions — should complete within a few seconds. LLM-assisted generation is allowed to take longer given the network round-trip involved, but the result should still feel responsive from the user's perspective.

    \item \textbf{Security:} The connected mode must operate in strict read-only mode. The application must have no capability to write, update, or delete data in any connected database. Connection credentials must not be stored or logged beyond the active session.

    \item \textbf{Maintainability:} The generation logic for each entity type must be isolated in its own service module, so that adding support for a new instrument type or updating a business rule does not require changes across the entire codebase.

    \item \textbf{Extensibility:} The system architecture must make it straightforward to add new entity types, new asset classes, or new output formats in the future without restructuring the core engine.

    \item \textbf{Usability:} The web interface must be usable by QA engineers who are not necessarily developers. Error messages must be clear, form inputs must be validated before submission, and the generated SQL output must be easy to copy or download.
\end{itemize}

\section{Global Architecture}
% ─────────────────────────────────────────────

\paragraph{}The requirements above point toward a classic full-stack web application architecture, with a clear separation between the user interface, the generation logic, the LLM integration, and the database access layer.

\begin{itemize}
    \item \textbf{Presentation layer (Frontend):} A React application that exposes the configuration forms, displays the generated SQL output, and handles user interaction. It communicates with the backend exclusively through a REST API.
    \item \textbf{Business logic layer (Backend):} A NestJS application that hosts all generation services — one per entity type. It receives configuration parameters from the frontend, orchestrates the generation sequence, optionally calls the Gemini API for realistic field values, and assembles the final SQL script.
    \item \textbf{LLM integration layer:} An optional component that sends structured prompts to the Google Gemini API and processes the responses. Its output feeds back into the generation services to enrich specific fields — security names, symbols, descriptions — with realistic financial content.
    \item \textbf{Data access layer:} Active only in connected mode. A read-only SQL Server connection that retrieves existing reference data from a live Longview instance, which the generation services then use as anchors for the new records they produce.
\end{itemize}

\section*{Conclusion}
\addcontentsline{toc}{section}{Conclusion}
% ─────────────────────────────────────────────

\paragraph{}This chapter covered the full analysis and design groundwork for the project. We chose Scrum as the development framework and organized the work into five sprints, each building on the previous one. A review of existing test data generation tools showed that none of them could handle the specificity of the Longview context, confirming the need for a purpose-built solution. From the functional and non-functional requirements, we derived a layered architecture built around a React frontend, a NestJS backend with entity-specific generation services, an optional Gemini API integration for data realism, and a read-only SQL Server connection for connected mode. The product backlog and sprint plan gave the implementation a clear roadmap. The next chapter covers Sprint 1 — the discovery phase where the existing SQL scripts were studied and the base generation engine was designed and built.
