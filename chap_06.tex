\subsubsection{GCC}

The GNU Compiler Collection (GCC), is an integrated compiler developed by the GNU Project. It supports a range of programming languages, including C, C++, Objective-C, and Fortran. GCC functions by taking source code written in these languages and generating binary executable programs in machine language. As an optimizing compiler, GCC supports various programming languages, hardware architectures, and operating systems. The compiler-driver \texttt{g++} is specifically used for C++ and is considered the de facto compiler for C/C++ on the Linux platform, including being the only compiler capable of compiling the Linux kernel. Additionally, we used \texttt{make}, a Unix tool that simplifies the process of building executable programs and libraries from source code. \texttt{make} reads files called Makefiles, which outline how to derive the target program, streamlining the build process \cite{gcc}.

To illustrate the dataset generation process and test it, we use Graph 6 as an example. We start by applying our approach to generate prefixes from Graph9, systematically traversing the graph to extract all possible prefixes of transitions. Once these prefixes are identified, we proceed to compile the complete dataset for the graph. This dataset includes each prefix, the set of possible choices for each prefix, and the best choice based on the optimal paths.

The results of this dataset generation process are summarized in table \ref{tab:dataset_graph6}. This table presents a detailed view of the prefixes, their corresponding possible choices, and the best choice for each prefix, showcasing how the approach is applied to a real-world graph and demonstrating its effectiveness in guiding state space exploration.

Each row of the table illustrates a distinct prefix— a sequence of transitions representing a sub-path of the graph. The prefixes are organized from the simplest, such as `a`, to more complex sequences like `abfca` and `dfcf`. For each prefix, the dataset enumerates:

\begin{itemize}
    \item \textbf{Possible Choices}: This column lists all the potential transitions that can follow from the given prefix. These choices are derived based on the structure of the graph and the paths available. For example, after the prefix `ab`, the possible choices are `a`, `b`, or `d`, indicating the different transitions that can be taken next.
    
    \item \textbf{Best Choice}: This column identifies the optimal transition based on the set of optimal paths previously computed. It reflects the transition that best aligns with the goal of optimizing coverage or achieving specific objectives. For instance, after the prefix `ab`, the best choice is `b`, as it represents the optimal transition from the choices available.
\end{itemize}


In summary, the dataset captures the essence of path exploration and optimization within Graph6. It helps in understanding the transition behavior at various stages of the graph and aids in guiding further exploration or decision-making processes in the context of the LTS.

We conducted a second analysis presented in table~\ref{tab:Randgraph_results}, using randomly generated deterministic LTS. For each experiment, we varied the number of nodes (column 1) from 1000 to 5000, and adjusted the number of arcs (column 2) accordingly. We also performed random selections from a fixed set of 1000 labels. Additionally, we specify the following features: total execution time in column 3, ILP size in column 4, average size of generated paths in column 5, number of optimal paths in column 6, and average size of optimal paths in the last column.

The column 4 indicates the size of linear programming algorithm, represented by the couple (r,c), where, $r$ is the number of rows and the $c$ is the number of columns. Notice that they represent respectively the number of labels aimed to be covered, and the number of generated paths. These metrics indicate the complexity of the linear programming problem generated.

\input{tables/tabRandSmall}

These results highlight the variability in computational complexity and path structures across different random graphs as the number of nodes and edges grows. Our proof of concept shows a memory overflow (i.e. MO) indicating that the memory limit was exceeded during the execution process, leading to an abrupt termination of the process. This interruption occurs after path generation, which highlights the limit of using ILP for very large LTS.

Our approach scales well with larger graphs up to a certain point. However, beyond this threshold, processing becomes impractical within a reasonable time frame (specifically, 1 hour), as indicated by occurrences of 'TO' (Time Out). The value 'MO' (Memory Overflow) indicates that the memory limit was exceeded during the execution process, leading to an abrupt termination of the process.

Therefore, with a very large number of initially generated paths, the percentage of optimal paths is significantly low, which is noteworthy.
These results offer important insights about our algorithm's performances and its limitations when handling large-scale graphs under parameterized random conditions.
